# Instagram Meme/Content Farm — Automated Posting Pipeline

**Date**: 2026-07-19
**Status**: Approved (pending user review of this document)

## Purpose

Run an Instagram account (broad relatable/internet-culture humor) that posts
daily with minimal ongoing human effort. User reviews and approves one week
of content at a time; after approval, publishing runs autonomously for the
rest of that week with no further check-ins.

## Hard Constraint: Official API Only

Automated posting uses the **Meta Graph API for Instagram** (Business/Creator
account, linked Facebook Page, Meta Developer App with a long-lived access
token). Unofficial routes (private-API libraries, browser automation) violate
Instagram's Terms of Service and risk shadowban or account termination with
no recourse. This project does not implement or support those routes.

The target IG account already exists as Business/Creator and is linked to a
Facebook Page. Meta App registration and token setup are in scope for this
project (user has not done this yet) — see "Setup" below.

## Content Model

- Broad, general relatable/internet-culture humor (daily life, work, phone
  habits, group chats — not tied to any specialist subject). One consistent
  voice/format across the account: 2026 engagement research (see "Content
  Strategy Basis" below) shows Instagram's algorithm rewards niche/format
  consistency even when the *topic* is broad, and penalizes accounts that
  post unrelated content day to day. "Niche" here means a consistent
  relatable-humor voice, not a subject-matter vertical.
- One post (static image) + one reel (short video) per day. Every one of
  the 14 weekly items has a `source`: `original | template | repost`.
  Reels are `original`/`template` only (4/3 split — see "Repost Sourcing
  (active)" below for why `repost` can't fill a reel slot). Posts are
  `repost`/`original`/`template` (3/2/2 split).
- Captions and hashtags: generated by Claude per item at content-generation
  time, written to optimize for DM-shareability ("would someone send this
  to a specific friend"), since 2026 engagement data ranks shares/sends per
  reach above likes as the dominant distribution signal.

### Content Sourcing

- **`original`** — Pollinations.ai (`image.pollinations.ai`), free, no API
  key, no signup required. Anonymous rate limit is roughly one request per
  15 seconds. Registering a free account raises the limit and drops the
  watermark; do this if anonymous limits become a problem. Reels: 2 images
  (setup shot + punchline shot) quick-cut together via `ffmpeg`, text hook
  overlaid only in the first ~3 seconds.
- **`template`** — a blank meme template fetched from Imgflip's free public
  template list (`api.imgflip.com/get_memes`, no key needed), with Claude
  writing top/bottom text rendered onto it via `ffmpeg`. Reels: the same
  rendered template image quick-cut through the same reel pipeline as
  `original`.
- **`repost`** — see "Repost Sourcing (active)" below.
- Instagram's 2026 "Originality Score" penalizes recycled/templated
  content; `original`/`template` produce fresh assets per item so neither
  is meaningfully exposed to that penalty. `repost` items are exposed to
  it (they're literally recycled content) — accepted as part of the same
  risk decision documented in "Repost Legal & Provenance Posture" below.

### Repost Sourcing history (Reddit, Imgur — dormant)

Six platforms were checked before landing on the active source below, each
ruled out for a distinct, verified reason:

| Platform | Reason unavailable |
|---|---|
| Reddit | Legacy Data API closed new-app registration to "valid moderation use case" only, per Reddit's own developer page (late 2025) |
| Imgur | Closed new API app registration entirely; registration page no longer works (December 2025) |
| Tenor | Stopped accepting new API clients January 2026; full service sunset June 30, 2026 |
| Giphy | Started charging for API access March 2026; ToS also requires a separate commercial agreement for "syndication" (posting content elsewhere) regardless |
| Tumblr | Registration may still be open, but the API agreement explicitly prohibits "syndicating or distributing Content in any manner, public or private" — the exact use case needed |
| Google Images (Custom Search API) | Doesn't host content, only indexes third-party sites with unknown/inconsistent copyright status; API itself is being discontinued January 1, 2027 |

`pipeline/reddit_source.py` and `pipeline/imgur_source.py` are both fully
implemented, tested, and reviewed — left in the codebase, unused, in case
either platform ever reopens this path. `pipeline/generate.py` does not
import or call either module.

### Repost Sourcing (active): apileague.com Random Meme API

**Adopted 2026-07-21, per explicit user decision, after the above six
platforms and direct scraping were both ruled out.** This is a
third-party service (`apileague.com`), not a first-party content
platform — its "Random Meme API" wraps another vendor ("Humor API") that
aggregates meme content from around the internet; example responses
observed during integration point at `i.redd.it`/`preview.redd.it` URLs,
meaning the underlying content is Reddit-hosted, reached via a proxy
rather than Reddit's own (unavailable) API. Verified against a real
request with a live key rather than guessed from docs:

- Endpoint: `GET https://api.apileague.com/retrieve-random-meme`
- Auth: `X-API-Key: <APILEAGUE_API_KEY>` header (free tier, self-service
  signup via `apileague.com/console/`, no approval queue)
- Response: flat JSON, no envelope — `{"description", "url", "type",
  "width", "height", "ratio"}`. No unique ID field (a dedupe ID is
  synthesized by hashing `url`) and **no NSFW/mature-content flag** — see
  content-safety note below.
- No listing/filter/search parameters confirmed beyond `max-age-days`; the
  endpoint returns exactly one random item per call, with no way to
  request a specific media type. Because of this, `repost` is restricted
  to **post slots only** — requesting a video-type item on demand isn't
  possible, and retrying repeatedly hoping for one would waste the daily
  quota for little payoff. Reel slots always use `original`/`template`.
- Free tier quota: 50 requests/day, shared across the whole account (not
  just this pipeline). `pick_unique_meme` retries up to a capped number
  of attempts against a locally-persisted seen-ID list
  (`content/apileague_seen.json`) before giving up on a slot.
- **No server-side NSFW filter exists for this API.** Every other source
  in this pipeline (original, template, and the dormant Reddit/Imgur
  modules) has a hard, code-enforced SFW filter; this one cannot, because
  the API doesn't expose a signal to filter on. Content safety for
  `repost` items is therefore **entirely dependent on the pipeline's
  existing weekly human review step** — nothing is posted without the
  user seeing it first in the review Artifact. This is a real, accepted
  gap in automated defense-in-depth, not an oversight.
- If `pick_unique_meme` exhausts its retry budget (all draws already
  seen) or the API call itself fails (bad key, quota exhausted, network
  error) — `pipeline.apileague_source.ApileagueSourceError` — that post
  slot falls back to `original`, same resilience pattern as every other
  producer failure in this pipeline.

### Repost Legal & Provenance Posture

Two accepted risks, both explicit user decisions for their own account,
recorded here so they aren't forgotten:

1. **Copyright/redistribution risk** — reposting images originally
   created by someone other than the uploading poster carries copyright
   exposure regardless of whether that poster is credited; the realistic
   worst case at this account's scale is a DMCA takedown / content
   removal request, not litigation. Per explicit user direction, `repost`
   items are not credited to any source. The user's stated reasoning:
   the account isn't monetized, which meaningfully lowers real-world
   enforcement likelihood even though it doesn't change the underlying
   legal exposure category.
2. **Third-party API terms** — apileague.com's terms provide content
   "as is" with no warranties and disclaim liability for content
   problems, permit caching for at most one hour, and don't disclose the
   underlying content's ownership/licensing chain. This pipeline retains
   fetched images indefinitely (committed to a public GitHub repo,
   posted permanently to Instagram), which exceeds the stated one-hour
   cache allowance. Per explicit user direction this is accepted
   knowingly — "no contractual obligation" concern, want to proceed.

It is not this project's place to silently reintroduce credit-as-
mitigation or additional filtering beyond what's documented above after
these decisions — but if either risk's calculus changes (e.g. the account
starts being monetized), this section should be revisited.

### Content Strategy Basis

Informed by July 2026 research on Instagram Reels/meme-account growth
(Buffer, TrueFuture Media, Ricky Spears — see plan doc for full citations):
sends-per-reach is the top distribution signal, first 1-3 seconds determine
whether a Reel gets distributed at all, niche/format consistency helps the
algorithm find an audience, and recycled/templated content is actively
penalized. These findings shaped the reel hook structure and the "broad
topic, consistent voice" niche framing above; they do not override the
user's explicit choices on niche breadth, content-source mix, or repost
crediting.

## Architecture

```
[Generate agent — weekly, e.g. Sun 07:00]
  → for each of the next 7 days, for post and reel (14 slots total):
      → look up this slot's source (original | template | repost — repost
        only ever assigned to post slots) from a fixed weekly rotation
      → produce the local asset + caption for that source:
          original → Pollinations.ai image(s) (+ ffmpeg reel composite)
          template → Imgflip blank template + Claude top/bottom text,
                      rendered via ffmpeg (+ ffmpeg reel composite)
          repost   → apileague.com random meme (post slots only), used
                      directly as the asset, no reel composite step
        each producer degrades independently on failure (retry, then skip
        that slot or fall back to original for repost) — never blocks
        the rest of the week's batch
      → asset pushed to a public GitHub repo (raw URL used as the public
        host Graph API requires — see "Asset Hosting")
      → new entry appended to content/queue.json
        (status: pending, scheduled_date: that day's date, source recorded)
  → publish/update the week's review page (Artifact)
  → push notification to user: "week's batch ready for review"

[User]
  → opens the Artifact review page once
  → approves or rejects each of the 14 items (or bulk-approves)
  → clicks Export, which downloads a copy of queue.json with each
    item's status patched to approved/rejected
  → hands that file back in a live Claude Code session, whenever
    convenient that week (no fixed deadline)

[Claude, in that live session]
  → commits the exported file over content/queue.json in the repo,
    replacing the all-pending version Generate created

[Publish agent — daily, two fires]
  → 12:00: find today's queue.json entry where type=post,
    scheduled_date=today, status=approved → publish via Graph API →
    status=posted
  → 20:00: same for type=reel
  → if today's entry is still status=pending (review/export/handoff
    hasn't happened yet, or the item was left unreviewed): skip it,
    log it, do NOT auto-post it, and do NOT re-prompt the user
```

### Why not automatic approval sync

The original design assumed Artifacts could persist approve/reject
decisions somewhere a scheduled routine could read back automatically.
Verified against the actual runtime capability contract while building
Task 14 (see the implementation plan): no such persisted-state capability
exists. The only two capabilities available are `downloads` (one-way,
viewer-initiated file save — not queryable by an agent) and `mcp` (calls a
connector the viewer has already connected — no GitHub connector exists in
this environment). Rather than guess at an unavailable API, the review
page uses `downloads` plus an Export button, and the human handoff (export
→ tell Claude → Claude commits) replaces the automatic sync step. This
still satisfies "approve once for the week, then hands-off" — the Publish
routines never need the user again once the exported file is committed —
it just moves the one required human action from "click approve in a page
a routine can see" to "click approve, then mention it once in chat."

### Why two phases + two daily publish fires

- Cron/scheduled agent runs cannot pause mid-run to wait on human input, so
  "generate" and "publish" must be separate scheduled runs gated by
  approval status stored in `content/queue.json`.
- Weekly generation + one-time weekly approval matches the user's explicit
  requirement: approve once for the week, then hands-off for the week.
- Posting time research (Buffer/Sprout Social 2026 datasets, ~9.6M posts):
  general engagement window is 11:00–19:00 with Tue–Thu strongest; reels
  specifically perform better in the 19:00–21:00 evening window. Splitting
  the daily publish into a ~12:00 image post and a ~20:00 reel post serves
  both windows instead of compromising on one slot for both formats.

## Components

- **`generate.py`** — weekly content orchestrator. For each of the 14
  weekly slots, dispatches to the right source producer (Pollinations,
  Imgflip+ffmpeg, or apileague.com), calls Claude for captions, pushes
  assets to the asset-hosting repo, writes `content/queue.json` entries,
  triggers the Artifact review page rebuild. Does not import or call
  `imgur_source.py` or `reddit_source.py`.
- **`image_gen.py`** — Pollinations.ai client for `original` images.
- **`reel_builder.py`** — `ffmpeg` quick-cut reel composer, used by
  `original` and `template` sources (not `repost`, which uses the fetched
  media directly and never fills a reel slot anyway).
- **`template_source.py`** — Imgflip blank-template fetch + `ffmpeg`
  top/bottom text rendering for `template` sources.
- **`apileague_source.py`** — apileague.com Random Meme API client:
  single-item fetch, dedupe-ID synthesis (hash of `url`, since the API
  provides none), retry-until-unseen, single-call media download for
  `repost` sources (post slots only).
- **`imgur_source.py`** / **`reddit_source.py`** — dormant/unused (see
  "Repost Sourcing history" for why). Both fully implemented, tested,
  and reviewed; kept in the codebase in case a platform reopens this path.
  `generate.py` calls neither.
- **`captions.py`** — Claude-backed caption/hashtag generation (all
  sources) and meme top/bottom text generation (`template` source only).
- **`content/queue.json`** — flat-file queue. One record per content item:
  ```json
  {
    "id": "uuid",
    "type": "post | reel",
    "source": "original | template | repost",
    "scheduled_date": "YYYY-MM-DD",
    "asset_url": "https://raw.githubusercontent.com/.../asset.jpg",
    "caption": "string",
    "hashtags": ["string", ...],
    "status": "pending | approved | rejected | posted | failed",
    "created_at": "ISO-8601",
    "posted_at": "ISO-8601 | null"
  }
  ```
  A flat file is sufficient at this volume (14 items/week); no database is
  needed.
- **`content/apileague_seen.json`** — flat list of previously-used meme
  dedupe IDs (sha256 of `url`) for the active `repost` source.
  **`content/imgur_seen.json`** / **`content/reddit_seen.json`** — dormant
  alongside their respective source modules.
- **`publish.py`** — Graph API client. Image flow: `POST /{ig-id}/media`
  with `image_url` + caption → `POST /{ig-id}/media_publish` with the
  returned `creation_id`. Reel flow: `POST /{ig-id}/media` with
  `video_url`, `media_type=REELS`, caption → poll `status_code` until
  `FINISHED` → `POST /{ig-id}/media_publish`. Supports a `--dry-run` flag
  that logs the exact calls it would make without hitting the live API.
  Identical regardless of source, active or dormant — `publish.py` only
  ever sees an `asset_url` and a `caption`, never a `source`.
- **Review Artifact page** — renders the week's 14 items (thumbnail/video
  preview, caption, hashtags, source) with approve/reject controls per
  item. Decisions persist in the browser (`localStorage`) and an Export
  button downloads a patched copy of `queue.json`; there is no capability
  that lets a scheduled routine read this page's decisions back
  automatically (see "Why not automatic approval sync" above) — the user
  hands the exported file back in a live chat session instead.
  (Implementation must load the `artifact-capabilities` skill before
  building this page.)
- **Two scheduled routines**: Generate (weekly) and Publish (daily, two
  fires: 12:00 and 20:00).
- **This repo, made public** — asset hosting. Graph API requires a publicly
  reachable URL for `image_url`/`video_url`; it does not accept direct file
  uploads. This project directory is initialized as a git repo and pushed
  to a public GitHub repo; `generate.py` commits generated assets here and
  raw.githubusercontent.com URLs serve as the public host. No secrets ever
  live in this repo (see Credentials & Secrets), so making it public is
  safe. One repo for both pipeline code and generated assets — no separate
  asset-only repo.

## Credentials & Secrets

Required: `IG_ACCESS_TOKEN` (long-lived Graph API token), `IG_BUSINESS_ID`
(the Instagram Business Account ID), `APILEAGUE_API_KEY` (free tier, for
`repost` sourcing), `ANTHROPIC_API_KEY`, and a GitHub token with push
access to the asset-hosting repo. All stored as environment variables /
the scheduled agent's secret store — never hardcoded, never committed to
the repo. (`REDDIT_CLIENT_ID`/`REDDIT_CLIENT_SECRET`/`IMGUR_CLIENT_ID` are
not required — both those source modules are dormant.) Long-lived IG
tokens expire (~60 days); the Publish agent must fail loudly and notify
the user on auth failure rather than fail silently, since a silently dead
token means an incomplete week appears successful but nothing actually
posts.

## Error Handling

- Pollinations 429 → backoff retry; if still failing, skip that slot for
  the day rather than blocking the rest of the week's batch.
- Imgflip/template render failure → skip that slot (falls back the same
  way as any other producer failure).
- apileague.com: retry budget exhausted (all draws already seen) or the
  API call itself fails (bad key, daily quota exhausted, network error) →
  that post slot falls back to `original` rather than failing the week's
  batch.
- Graph API publish failure → log, leave `status=approved` (so the next
  Publish run retries it), never silently drop an approved item.
- Expired/invalid token → Publish run fails loudly, notifies user.
- Item still `pending` at its scheduled publish time → skipped, logged,
  never force-posted, never re-prompts (review window already passed).

## Testing

- `publish.py --dry-run` — validates the Graph API call sequence without
  posting live.
- `content/queue.json` schema validated on every read/write.
- No automated test can safely post to the real account, hit the live
  Pollinations/Imgflip/apileague.com/Anthropic APIs, or perform a real
  `git push` — every external call is injected via a `session`/`runner`/
  `client` parameter and replaced with a fake in tests.
- A manual smoke test (one real image post, verified in the app) is
  required before the pipeline is trusted to run unattended for a full
  week.
